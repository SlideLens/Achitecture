# Architecture diagrams (C4)

Levels C1–C3 follow the [C4 model](https://c4model.com/); the classic level 4 (“code”) is replaced by a data-model ERD — SQLAlchemy models are written from it. Plus a Review pipeline diagram and a sequence for the key flow. Terms — per [CONTEXT.md](../CONTEXT.md); decisions — per [ADR](../adr/).

> The same diagrams also exist as sources in [docs/diagrams/](diagrams/) ([d2](https://d2lang.com/) format) for building a polished PDF report — see [report/](../report/). Mermaid below is a render-independent variant readable directly on GitHub.

```mermaid
    user -- "upload deck/pitch,<br/>read report [HTTPS]" --> slidelens
    slidelens -- "prompts + images / JSON [HTTPS]" --> vlm

```
*Audience:* stakeholder — who uses the system and what it talks to externally. Whisper and python-pptx are omitted: they are internal libraries, not external services. Observability is dashed — a dev/ops plane, not a runtime dependency (if Langfuse is down, the Review continues).

## C2 — Containers

Prod: Docker Compose on a VPS. The built SPA is baked into the backend image (`./static`) and served by FastAPI itself; Caddy on a single domain terminates TLS and reverse-proxies all traffic to app (no CORS) — see [DEPLOY.md](DEPLOY.md).

```mermaid
flowchart TB
    user["👤 User (browser)"]

    subgraph vps["VPS · Docker Compose"]
        caddy["🔀 Caddy<br/>HTTPS/TLS, single domain:<br/>reverse-proxy everything to app"]
        spa["🖥️ Frontend · React SPA<br/>Vite + TS + Tailwind + TanStack Query<br/>(dist baked into image → ./static)"]
        api["⚙️ Backend · FastAPI<br/>JSON API + JWT + limits +<br/>file intake, report and static serving"]
        worker["🔧 Worker · review pipeline<br/>ARQ (same image as app)<br/>ingest → analysis → report"]
        db[("🗄️ PostgreSQL<br/>User, Review, Finding,<br/>FileAsset, Event")]
        redis[("⚡ Redis<br/>ARQ queue")]
        files[("📁 Files: disk → S3/R2<br/>originals, PNGs, reports, fixed.pptx")]
    end

    vlm["☁️ Claude API (vision)"]
    email["✉️ SMTP"]

    user -- "HTTPS" --> caddy
    caddy -- "TLS reverse proxy" --> api
    api -- "serves SPA from ./static" --> spa
    api -- "SQL" --> db
    api -- "enqueue job" --> redis
    api -- "store/serve" --> files
    redis --> worker
    worker -- "vision [HTTPS]" --> vlm
    worker -- "findings, score, status" --> db
    worker -- "artifacts" --> files
    worker -- "report ready" --> email
```
One origin for API and static → no CORS and no nginx ([ADR 0004](../adr/0004-stack-fastapi-react.md)). The Review runs in the worker, not in the HTTP request ([ADR 0003](../adr/0003-async-review-worker.md)).

*Audience:* developer/DevOps — what actually deploys and where data, queue, and files live.

## C3 — Backend and pipeline components

*Audience:* developer — how the web layer and pipeline are decomposed, and where the “app knows DB ↔ core is pure” boundary sits. All VLM calls go through the single `LLMClient` ([ADR 0002](../adr/0002-vlm-pipeline-hybrid-analyzers.md)).

## Review pipeline

Ten steps from uploaded Deck to Report. Failure of any analyzer does not kill the Review — a partial report beats `failed` ([ADR 0002](../adr/0002-vlm-pipeline-hybrid-analyzers.md)).

```mermaid
flowchart TB
    start(["Review: status = processing"]) --> ingest

    subgraph s1["1. Ingest"]
        ingest["DeckIngestor<br/>PPTX/PDF → PDF (LibreOffice) → PNG (pdf2image)<br/>+ text extraction (python-pptx)"]
        transcribe["Transcriber<br/>audio → WAV → faster-whisper<br/>→ transcript + DeliveryMetrics"]
        excel["Excel parse<br/>(if attached)"]
    end

    subgraph s2["2. Analysis (VLM = Claude vision)"]
        slide["SlideAnalyzer<br/>hierarchy, typography, readability"]
        zoom["ZoomAgent<br/>suspicious regions → crop ×2 → close-up"]
        deck["DeckAnalyzer<br/>consistency, narrative, duplicates"]
        chart["ChartChecker<br/>truncated axes, pie≠100%, Excel reconcile"]
        cross["CrossModalAnalyzer<br/>speech ↔ slides, deck recommendations"]
    end

    subgraph s3["3. Assemble result"]
        agg["Aggregator + DeckScorer<br/>dedupe, prioritize, score 0–100"]
        annot["Annotator<br/>Pillow: bbox frames on PNG"]
        fixer["PptxFixer<br/>contrast/alignment/font → fixed.pptx"]
        report["ReportBuilder + PdfExporter<br/>HTML view-model + PDF (weasyprint)"]
    end

    ingest --> slide
    ingest -.-> transcribe
    ingest -.-> excel
    slide --> zoom
    zoom --> deck
    slide --> chart
    excel -.-> chart
    transcribe -.-> cross
    deck --> agg
    chart --> agg
    cross -.-> agg
    agg --> annot
    agg --> fixer
    annot --> report
    fixer --> report
    report --> done(["status = done · email user"])
    ingest -->|"render error"| fail(["status = failed + fail_reason,<br/>fallback: attach PDF"])
```
*Audience:* developer — step order and pipeline modules; dashed lines are optional branches (audio/Excel are not always attached).

## Level 4 — Data model (ERD)

Instead of classic C4 “code”: SQLAlchemy models are written from this schema. `FindingRow` mirrors the pipeline pydantic `Finding` ([ADR 0001](../adr/0001-pipeline-pure-library.md)).

```mermaid
erDiagram
    USER ||--o{ REVIEW : "creates"
    REVIEW ||--o{ FINDING : "contains"
    REVIEW ||--o{ FILE_ASSET : "stores"
    REVIEW ||--o| REHEARSAL : "phase 4"
    USER ||--o{ EVENT : "emits"

    USER {
        uuid id PK
        string email UK
        string password_hash
        string plan "free | paid"
        int free_reviews_left "limit: 2"
        bool email_verified
        datetime created_at
    }
    REVIEW {
        uuid id PK
        uuid user_id FK
        string status "queued | processing | done | failed"
        int score "0-100, nullable"
        string fail_reason "nullable"
        string deck_filename "uploaded file name"
        int n_slides "nullable until successful ingest"
        json delivery_metrics "nullable, from CrossModalAnalyzer when has_audio"
        bool has_audio
        bool has_data
        datetime created_at
        datetime finished_at "nullable"
    }
    FINDING {
        uuid id PK
        uuid review_id FK
        int slide_num "nullable = deck level"
        string category "TYPOGRAPHY|HIERARCHY|READABILITY|CONSISTENCY|CHART|NARRATIVE|SPEECH_MISMATCH|DELIVERY"
        string severity "CRITICAL | MAJOR | MINOR"
        string title "<=80"
        text description
        text fix_suggestion
        json bbox "normalized 0..1, nullable"
        uuid screenshot_asset_id FK "annotated PNG (FILE_ASSET), nullable"
        bool auto_fixable
        bool auto_fixed
        string source "analyzer name"
        bool user_flag "👎 junk finding"
        bool user_like "👍 useful finding"
    }
    FILE_ASSET {
        uuid id PK
        uuid review_id FK
        string kind "deck_original|slide_png|annotated_png|fixed_pptx|audio|data_xlsx|report_pdf"
        string storage_path
        int size_bytes
        int version "increments on regen (fixed_pptx per apply_fix)"
        datetime expires_at "auto-delete"
    }
    EVENT {
        uuid id PK
        uuid user_id FK "nullable"
        string name
        json properties
        datetime created_at
    }
    REHEARSAL {
        uuid id PK
        uuid review_id FK
        string audio_path
        json slide_timings "switch timestamps"
        json delivery_metrics
        int attempt_num "cross-run dynamics"
        datetime created_at
    }
```

Model notes:
- **Score and Findings are persisted** — the Report (`ReportOut`) is assembled from `FINDING` + `FILE_ASSET`, stored, and served via `GET /reviews/{id}/report`.
- **`bbox`** — normalized coordinates `0..1` (dpi-independent); used to draw frames on PNG.
- **`FILE_ASSET.expires_at`** — privacy (US-8): a periodic job deletes expired files from Storage.
- **`REVIEW.delivery_metrics`** — Delivery for a pitch Recording attached to the Review (MVP, US-2/US-4). Separate from `REHEARSAL`, which stays empty for phase 4 (in-browser recording with precise `SlideTiming`, cross-run dynamics).
- **`user_flag` (👎) / `user_like` (👍)** — mutually exclusive votes; flow into Langfuse as a dataset for prompt iteration ([ADR 0007](../adr/0007-three-layer-observability.md)). Targeted autofix — `POST /findings/{id}/apply_fix` (set `auto_fixed`, regen `fixed.pptx` from the original).
- **`REHEARSAL`** — empty stub for phase 4 ([ADR 0005](../adr/0005-crossmodal-delivery-analysis.md)); not populated in MVP, but created so the feature does not break the schema.

*Audience:* developer — SQLAlchemy models are written directly from this ERD.

## Sequence: user path (upload → report)

```mermaid
sequenceDiagram
    actor U as User
    participant API as FastAPI
    participant Q as Worker (pipeline)
    participant C as Claude API / Whisper
    participant S as DB + files
    participant M as Email

    U->>API: Register → confirm email → login
    U->>API: POST /reviews (deck + audio? + xlsx?)
    API->>API: check limit (free_reviews_left)
    API->>S: Review(status=queued), store files
    API-->>U: 202 — "review started" (cabinet)
    API->>Q: enqueue job

    activate Q
    Q->>Q: render slides to PNG
    Q->>C: Whisper — transcript with timestamps
    loop per slide + zooms
        Q->>C: VLM analysis (hierarchy, typography, charts)
        C-->>Q: Findings (JSON)
    end
    Q->>C: cross-slide analysis + speech ↔ slides
    Q->>Q: aggregate, score, annotate, autofix → fixed.pptx
    Q->>S: findings, score, artifacts, status=done
    deactivate Q
    Q->>M: "report ready"
    M-->>U: email with link

    U->>API: GET /reviews/{id}/report
    API-->>U: score, findings per slide, "charts / delivery / speech↔slides" blocks
    U->>API: download PDF / fixed.pptx
    API-->>U: files (owner check)
```
*Audience:* developer — happy path of the product nail; the `failed` branch (render error) and ownership errors (404) are documented in [api/openapi.yaml](../api/openapi.yaml) and omitted here.
